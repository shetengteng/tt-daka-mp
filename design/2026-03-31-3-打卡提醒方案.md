# DaKa 打卡提醒 - 技术方案设计

## 1. 需求概述

用户可以设置每日打卡提醒，在指定时间收到微信服务通知，提醒完成当天的打卡任务。

## 2. 微信小程序推送能力分析

### 2.1 可选方案对比

| 方案 | 说明 | 限制 | 推荐 |
|------|------|------|------|
| **订阅消息** | 用户主动订阅后，可发送模板消息 | 每次订阅只能发一条（一次性）或长期订阅（需审核） | 推荐 |
| ~~模板消息~~ | 已废弃（2020年1月起） | 不可用 | - |
| **统一服务消息** | 通过公众号关联发送 | 需要公众号 + 小程序关联 | 备选 |
| **本地通知** | 小程序不支持 | 不可用 | - |

### 2.2 推荐方案：订阅消息

微信订阅消息分为两种：

- **一次性订阅**：用户每次点击授权，只能发送一条消息。适合"下次提醒"场景。
- **长期订阅**：需要在微信后台申请，审核通过后可持续发送。适合"每日提醒"场景，但**仅限政务、医疗等特定类目**，普通工具类小程序**很难通过审核**。

**结论**：DaKa 作为工具类小程序，大概率只能使用**一次性订阅消息**。

## 3. 一次性订阅消息方案

### 3.1 交互流程

```
用户打开设置页
    │
    ▼
点击「打卡提醒」开关 → 弹出微信订阅授权弹窗
    │
    ├── 用户同意 → 记录订阅次数 +1，开关变为开启
    │
    └── 用户拒绝 → 开关保持关闭
    
每次打卡完成后
    │
    ▼
弹出提示：「是否订阅明天的打卡提醒？」
    │
    ├── 用户同意 → 调用 wx.requestSubscribeMessage，订阅次数 +1
    │
    └── 用户跳过 → 不订阅
```

### 3.2 消息发送时机

| 触发场景 | 说明 |
|----------|------|
| 用户设置的提醒时间（如 20:00） | 云函数定时任务检查，对有剩余订阅次数且当天未完成打卡的用户发送提醒 |
| 每日固定时间（如 21:00） | 兜底提醒，对当天有未完成项目的用户发送 |

### 3.3 技术架构

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  小程序前端   │────▶│  EMAS 云函数       │────▶│  微信订阅消息 API │
│             │     │  (定时触发器)       │     │                 │
│ 订阅授权     │     │  检查未打卡用户     │     │  发送模板消息     │
│ 记录订阅次数  │     │  消耗订阅次数       │     │                 │
└─────────────┘     └──────────────────┘     └─────────────────┘
```

### 3.4 数据库设计

在 `dk-users` 表中新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| reminderEnabled | boolean | 是否开启提醒 |
| reminderTime | string | 提醒时间，如 "20:00" |
| subscribeCount | number | 剩余订阅次数 |
| lastSubscribeTime | string | 最后一次订阅时间 |

### 3.5 前端实现要点

```javascript
// 请求订阅消息
wx.requestSubscribeMessage({
  tmplIds: ['模板ID'],
  success(res) {
    if (res['模板ID'] === 'accept') {
      // 用户同意，更新 subscribeCount + 1
    }
  }
})
```

**关键点**：
- `tmplIds` 需要在微信后台「订阅消息」中创建模板并获取 ID
- 每次调用 `requestSubscribeMessage` 都会弹出授权弹窗
- 用户可以勾选「总是保持以上选择」，之后不再弹窗

### 3.6 云函数定时任务

```javascript
// 伪代码：每日定时检查并发送提醒
async function sendReminders() {
  const now = dayjs()
  const currentTime = now.format('HH:mm')
  
  // 查找需要提醒的用户
  const users = await db.collection('dk-users').find({
    reminderEnabled: true,
    reminderTime: currentTime,
    subscribeCount: { $gt: 0 }
  })
  
  for (const user of users) {
    // 检查今天是否有未完成的打卡
    const hasUndone = await checkUndoneProjects(user.accountId)
    if (!hasUndone) continue
    
    // 发送订阅消息
    await cloud.openapi.subscribeMessage.send({
      touser: user.openid,
      templateId: '模板ID',
      data: {
        thing1: { value: '你还有打卡任务未完成' },
        time2: { value: now.format('YYYY-MM-DD HH:mm') }
      }
    })
    
    // 消耗订阅次数
    await db.collection('dk-users').updateOne(
      { _id: user._id },
      { $inc: { subscribeCount: -1 } }
    )
  }
}
```

## 4. 替代方案：无服务端的本地提醒

如果暂时不接入云函数，可以使用以下轻量方案：

### 4.1 打卡完成后引导订阅

每次用户完成打卡后，弹出提示引导订阅明天的提醒。这样无需定时任务，用户主动触发。

### 4.2 利用小程序「周期性更新」

微信小程序支持 `wx.setBackgroundFetchToken`，可以在后台周期性拉取数据并更新小程序。但这不能发送通知，只能更新数据。

### 4.3 公众号关联

如果有关联的公众号，可以通过公众号模板消息发送提醒，不受订阅次数限制。但需要用户关注公众号。

## 5. 实施建议

### Phase 1（MVP）

1. 在设置页保留「打卡提醒」开关和时间选择 UI
2. 开关点击时调用 `wx.requestSubscribeMessage` 请求一次性订阅
3. 每次打卡完成后引导用户订阅明天的提醒
4. 将订阅状态存储到 `dk-users` 表

### Phase 2（完整版）

1. 创建 EMAS 云函数定时任务
2. 在微信后台申请订阅消息模板
3. 实现定时检查 + 消息发送逻辑
4. 添加订阅次数管理和提示

### Phase 3（增强）

1. 尝试申请长期订阅（如果类目允许）
2. 关联公众号作为备用通知渠道
3. 添加更丰富的提醒内容（如连续打卡天数、即将断签提醒等）

## 6. 注意事项

- 一次性订阅消息的模板需要在微信后台创建，审核周期约 1-3 天
- 用户拒绝授权后，短期内再次请求可能被微信限制
- 建议在合适的时机（如打卡完成后）引导订阅，避免频繁打扰
- 测试时需要使用真机，开发者工具不支持订阅消息弹窗
