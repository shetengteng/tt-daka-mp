# 离线缓存设计方案（Local-First）

## 背景

EMAS 免费版每日有请求额度限制（`PrePayResourceExhausted`），超限后所有数据库操作失败。当前系统在云服务不可用时，用户无法打卡、无法查看数据。

**核心思路**：所有操作先写本地 localStorage，UI 立即响应；云端同步作为后台异步任务，可批量合并、延迟执行，大幅减少 API 调用次数。

## 存储结构

### localStorage Key 设计

```
dk_{accountId}/
├── cache_projects        → 项目列表快照（projectStore.persist）
├── cache/
│   ├── today_{YYYY-MM-DD} → 当日打卡记录（recordStore.persist）
│   └── month_{YYYY-MM}   → 月度打卡记录（getRecordsByMonth 缓存）
├── pending/
│   └── ops               → 待同步操作队列
└── meta/
    └── dataVersion        → 数据版本号（时间戳乐观锁）
```

> **注意**：用户信息（nickname/avatar/totalDays）通过 `userStore` 在内存中管理，未持久化到 localStorage。

**实际 Key 示例**：

| Key | 说明 | 示例值 |
|-----|------|--------|
| `dk_wx_xxx_cache_projects` | 项目列表 | `[{_id, name, icon, ...}]` |
| `dk_wx_xxx_cache_today_2026-04-05` | 今日打卡 | `[{_id, projectId, date, ...}]` |
| `dk_wx_xxx_cache_month_2026-04` | 4月打卡 | `[{_id, projectId, date, ...}]` |
| `dk_wx_xxx_pending_ops` | 操作队列 | `[{id, op, data, ...}]` |
| `dk_wx_xxx_meta_dataVersion` | 数据版本号 | `1712300000000` |

## 缓存触发时机

### 写缓存时机

| 时机 | 写入什么 | 说明 |
|------|---------|------|
| 云端查询成功 | 对应查询的完整结果 | projects / records / user |
| 本地写操作 | 增量更新对应缓存 | 打卡→追加 todayRecords |
| 同步成功 | 更新 meta.dataVersion | touchCloudVersion 更新版本号 |

### 读缓存时机

| 时机 | 读什么 | 条件 |
|------|--------|------|
| 页面 onShow | projects + todayRecords | 始终先读本地 |
| 日历切换月份 | month_{YYYY-MM} | 本地有则直接用 |
| 云端请求失败 | 降级读本地所有缓存 | Quota/网络错误 |

### 同步触发时机

| 时机 | 行为 | 说明 |
|------|------|------|
| App onLaunch/onShow | 检查队列，有则同步 | 启动或回到前台 |
| 下拉刷新 | 强制同步 + 全量拉取 | 用户主动刷新 |
| 写操作后（延迟） | 300ms 后尝试同步 | 防抖，合并短时间多次操作 |
| 定时器 | 30s 轮询检查 | 静默后台同步 |

## Local-First 写入策略（节省 API 次数）

### 核心思路

传统模式：每次打卡 → 立即调云端 → 1 次 API  
Local-First：N 次打卡 → 本地即时响应 → 批量同步 → 可能只需 1-2 次 API

### 批量合并规则

```
场景1：对同一项目 先打卡后取消
  add(proj_A, 04-05) + remove(proj_A, 04-05) → 抵消，无需同步

场景2：同一时间段多个项目打卡
  add(proj_A) + add(proj_B) + add(proj_C) → 保持 3 条，无法合并（不同文档）

场景3：补打卡多天
  add(proj_A, 04-01) + add(proj_A, 04-02) + add(proj_A, 04-03) → 保持 3 条
```

**合并算法**：

```js
function mergeOps(ops) {
  const merged = []
  const cancelMap = new Map() // key: collection_projectId_date

  // 反向扫描，标记 add+remove 抵消对
  for (let i = ops.length - 1; i >= 0; i--) {
    const op = ops[i]
    const key = `${op.collection}_${op.data?.projectId}_${op.data?.date || ''}`

    if (op.op === 'remove') {
      cancelMap.set(key, i)
    } else if (op.op === 'add' && cancelMap.has(key)) {
      ops[i].merged = true
      ops[cancelMap.get(key)].merged = true
      cancelMap.delete(key)
    }
  }

  return ops.filter(op => !op.merged)
}
```

### 效果对比

| 场景 | 传统模式 | Local-First | 节省 |
|------|---------|-------------|------|
| 打卡 3 个项目 | 3 次 API | 3 次（无法合并） | 0% |
| 打卡后反悔取消 | 2 次 API | 0 次（抵消） | 100% |
| 日内反复打卡取消 5 次 | 5 次 API | 0-1 次 | 80-100% |
| 补打卡一周 7 天 | 7 次 API | 7 次（批量执行） | 0%（但 UI 无延迟） |
| 额度用完后打卡 | 失败 | 本地成功，次日同步 | 用户无感 |

## 版本号机制（减少无效拉取）

### 问题

后台静默拉取云端数据时，即使数据没有变化也会消耗 API 调用次数。需要一种机制判断"云端数据是否有更新"，没有更新则跳过拉取。

### 方案

在 `dk-users` 表中增加 `dataVersion` 字段（整数），每次写操作（打卡/取消/新建项目等）成功同步到云端时递增。本地缓存记录上次同步的版本号，对比后决定是否需要全量拉取。

### 数据结构变更

```js
// dk-users 表增加字段
{
  // ...原有字段
  dataVersion: 42,               // 数据版本号，每次写操作后 +1
}

// localStorage 缓存增加版本号
{
  data: [...],
  timestamp: 1712300000000,
  source: 'cloud',
  dataVersion: 42,              // 记录缓存对应的版本号
}
```

### 版本号流程

```
┌─────────────────────────────────────────────┐
│                写操作同步成功                   │
│   toggleDaka / createProject / retroactive   │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
         ┌──────────────────┐
         │ user.dataVersion │
         │      += 1        │
         └────────┬─────────┘
                  │
                  ▼
         ┌──────────────────┐
         │ 本地缓存          │
         │ cacheVersion = N │
         └──────────────────┘

──────── 下次读操作时 ────────

         ┌──────────────────────────────┐
         │ 1. 查询 user.dataVersion (1次)│
         └──────────┬───────────────────┘
                    │
           ┌────────┴────────┐
           │                  │
     dataVersion ==      dataVersion !=
     cacheVersion        cacheVersion
           │                  │
           ▼                  ▼
     ┌──────────┐    ┌──────────────┐
     │ 跳过拉取  │    │ 全量拉取数据  │
     │ 省 2+ 次  │    │ 更新缓存     │
     │ API 调用  │    │ 更新版本号   │
     └──────────┘    └──────────────┘
```

### 版本号检查接口

```js
// utils/version-check.js

import { db, COLLECTIONS } from '@/cloud-emas/database/database'
import { getLocal, setLocal, getStoreKey } from './local-store'

export async function checkDataVersion(accountId) {
  const versionKey = getStoreKey(accountId, 'meta', 'dataVersion')
  const localVersion = getLocal(versionKey)?.data || 0

  try {
    const res = await db.collection(COLLECTIONS.USERS)
      .where({ accountId })
      .get()
    const user = (res.data || [])[0]
    const cloudVersion = user?.dataVersion || 0

    if (cloudVersion === localVersion && localVersion > 0) {
      return { needRefresh: false, version: localVersion }
    }

    return { needRefresh: true, version: cloudVersion }
  } catch (e) {
    return { needRefresh: false, version: localVersion }
  }
}

export function updateLocalVersion(accountId, version) {
  const versionKey = getStoreKey(accountId, 'meta', 'dataVersion')
  setLocal(versionKey, version)
}

export async function incrementCloudVersion(accountId) {
  try {
    const res = await db.collection(COLLECTIONS.USERS)
      .where({ accountId })
      .get()
    const user = (res.data || [])[0]
    if (!user) return

    const newVersion = (user.dataVersion || 0) + 1
    await db.collection(COLLECTIONS.USERS).doc(user._id).update({
      dataVersion: newVersion,
    })
    return newVersion
  } catch (e) {
    // 版本号更新失败不影响主流程
  }
}
```

### 版本号使用场景

| 场景 | 行为 | API 调用 |
|------|------|---------|
| 页面 onShow（有缓存） | 查 user.dataVersion → 对比 → 跳过/拉取 | 1 次（仅查 version） |
| 页面 onShow（无缓存） | 直接全量拉取 | 2-3 次 |
| 打卡同步成功 | dataVersion += 1 | 1 次额外 update |
| 下拉刷新 | 强制拉取，不检查版本 | 正常次数 |

### 效果

| 场景 | 无版本号 | 有版本号 | 节省 |
|------|---------|---------|------|
| 切换 Tab 4 次（数据无变化） | 8 次 API（每次 2 查询） | 4 次 API（每次 1 查 version） | 50% |
| App 启动恢复（数据无变化） | 2-3 次 | 1 次 | 50-66% |
| 一天内切 Tab 20 次 | 40 次 | 20 次 | 50% |

### 注意事项

| 注意点 | 说明 |
|--------|------|
| 版本号递增成本 | 每次写操作多 1 次 update，但写操作频率远低于读操作 |
| 首次使用 | user.dataVersion 不存在时视为 0，触发全量拉取 |
| 离线操作 | 本地写操作不递增 cloudVersion，同步成功后才递增 |
| 版本号溢出 | 整数递增，不存在溢出问题（JS 安全整数上限 2^53） |

```js
const PREFIX = 'dk_'

export function getStoreKey(accountId, ...parts) {
  return `${PREFIX}${accountId}_${parts.join('_')}`
}

export function setLocal(key, data) {
  uni.setStorageSync(key, JSON.stringify({
    data,
    timestamp: Date.now(),
    source: 'local',
  }))
}

export function getLocal(key) {
  const raw = uni.getStorageSync(key)
  if (!raw) return null
  return JSON.parse(raw)
}

export function removeLocal(key) {
  uni.removeStorageSync(key)
}

// 读缓存快捷方法
export function getCachedProjects(accountId) {
  const entry = getLocal(getStoreKey(accountId, 'cache', 'projects'))
  return entry?.data || null
}

export function setCachedProjects(accountId, projects) {
  setLocal(getStoreKey(accountId, 'cache', 'projects'), projects)
}

export function getCachedTodayRecords(accountId, date) {
  const entry = getLocal(getStoreKey(accountId, 'cache', `today_${date}`))
  return entry?.data || null
}

export function setCachedTodayRecords(accountId, date, records) {
  setLocal(getStoreKey(accountId, 'cache', `today_${date}`), records)
}
```

```js
// utils/pending-ops.js

import { getLocal, setLocal, getStoreKey } from './local-store'

function _getKey(accountId) {
  return getStoreKey(accountId, 'pending', 'ops')
}

export function getPendingOps(accountId) {
  const entry = getLocal(_getKey(accountId))
  return entry?.data || []
}

export function addPendingOp(accountId, op) {
  const ops = getPendingOps(accountId)
  ops.push({
    ...op,
    id: `op_${Date.now()}_${Math.random().toString(36).slice(2, 6)}`,
    timestamp: Date.now(),
    retryCount: 0,
    merged: false,
  })
  setLocal(_getKey(accountId), ops)
}

export function removePendingOp(accountId, opId) {
  const ops = getPendingOps(accountId).filter(o => o.id !== opId)
  setLocal(_getKey(accountId), ops)
}

export function clearPendingOps(accountId) {
  setLocal(_getKey(accountId), [])
}

export function getPendingCount(accountId) {
  return getPendingOps(accountId).length
}
```

### 模块三：SyncManager（同步管理器）

```js
// utils/sync-manager.js

import { getPendingOps, removePendingOp } from './pending-ops'
import { db, COLLECTIONS } from '@/cloud-emas/database/database'
import { isQuotaError } from '@/cloud-emas/database/error'

let _syncing = false
let _syncTimer = null

export function startSyncLoop() {
  if (_syncTimer) return
  _syncTimer = setInterval(() => {
    syncPendingOps()
  }, 30 * 1000)
}

export function stopSyncLoop() {
  if (_syncTimer) {
    clearInterval(_syncTimer)
    _syncTimer = null
  }
}

export async function syncPendingOps(accountId) {
  if (_syncing || !accountId) return { synced: 0, remaining: 0 }

  let ops = getPendingOps(accountId)
  if (ops.length === 0) return { synced: 0, remaining: 0 }

  // 合并抵消操作
  ops = mergeOps(ops)

  _syncing = true
  let syncedCount = 0

  for (const op of ops) {
    try {
      switch (op.op) {
        case 'add':
          await db.collection(op.collection).add(op.data)
          break
        case 'remove':
          await db.collection(op.collection).where(op.where).remove()
          break
        case 'update':
          await db.collection(op.collection).doc(op.docId).update(op.data)
          break
      }
      removePendingOp(accountId, op.id)
      syncedCount++
    } catch (e) {
      if (isQuotaError(e)) break
      op.retryCount++
      if (op.retryCount >= 3) {
        removePendingOp(accountId, op.id)
      }
      break
    }
  }

  _syncing = false

  const remaining = getPendingOps(accountId).length
  if (syncedCount > 0) {
    uni.showToast({ title: `已同步 ${syncedCount} 条离线记录`, icon: 'none' })
  }

  return { synced: syncedCount, remaining }
}

function mergeOps(ops) {
  const cancelMap = new Map()

  for (let i = ops.length - 1; i >= 0; i--) {
    const op = ops[i]
    if (op.collection !== COLLECTIONS.RECORDS) continue

    const key = `${op.data?.projectId}_${op.data?.date || op.where?.date}`

    if (op.op === 'remove') {
      cancelMap.set(key, i)
    } else if (op.op === 'add' && cancelMap.has(key)) {
      ops[i].merged = true
      ops[cancelMap.get(key)].merged = true
      cancelMap.delete(key)
    }
  }

  return ops.filter(op => !op.merged)
}
```

## 改造示例：toggleDaka

```js
export async function toggleDaka(projectId, currentChecked) {
  const accountId = await requireAccountId()
  const today = formatDate(new Date())

  if (currentChecked) {
    // 取消打卡 → 立即更新本地
    removeCachedTodayRecord(accountId, today, projectId)
    addPendingOp(accountId, {
      op: 'remove',
      collection: COLLECTIONS.RECORDS,
      where: { accountId, projectId, date: today },
      data: { projectId, date: today },
    })
    // 异步同步，不阻塞
    debouncedSync(accountId)
    return { success: true, action: 'cancel' }
  }

  // 打卡 → 立即写入本地
  const now = dayjs().format('YYYY-MM-DD HH:mm:ss')
  const record = {
    accountId, projectId, date: today,
    completedAt: now, isRetroactive: false, createTime: now,
    _id: `local_${Date.now()}`,
  }

  addCachedTodayRecord(accountId, today, record)
  addPendingOp(accountId, {
    op: 'add',
    collection: COLLECTIONS.RECORDS,
    data: record,
  })
  debouncedSync(accountId)
  return { success: true, action: 'check', record }
}

// 防抖同步：300ms 内多次操作只触发一次
let _syncTimeout = null
function debouncedSync(accountId) {
  if (_syncTimeout) clearTimeout(_syncTimeout)
  _syncTimeout = setTimeout(() => syncPendingOps(accountId), 300)
}
```

## 改造示例：getProjectList

```js
export async function getProjectList() {
  const accountId = await requireAccountId()
  const today = formatDate(new Date())

  // 先返回本地缓存（如果有）
  const cachedProjects = getCachedProjects(accountId)
  const cachedRecords = getCachedTodayRecords(accountId, today)

  if (cachedProjects) {
    // 有缓存 → 先返回本地，后台静默更新
    silentCloudFetch(accountId, today)
    return {
      success: true,
      list: cachedProjects,
      todayRecords: cachedRecords || [],
      fromCache: true,
    }
  }

  // 无缓存 → 走云端
  try {
    const result = await fetchFromCloud(accountId, today)
    setCachedProjects(accountId, result.list)
    setCachedTodayRecords(accountId, today, result.todayRecords)
    return result
  } catch (error) {
    return { success: false, list: [], todayRecords: [] }
  }
}

async function silentCloudFetch(accountId, today) {
  try {
    const result = await fetchFromCloud(accountId, today)
    setCachedProjects(accountId, result.list)
    setCachedTodayRecords(accountId, today, result.todayRecords)
    // 通知 store 刷新（如果数据有变化）
    const dakaStore = useDakaStore()
    dakaStore.setProjects(result.list)
    dakaStore.setTodayRecords(result.todayRecords)
    dakaStore.markFresh()
  } catch (e) {
    // 静默失败，不影响用户
  }
}
```

## localStorage 容量管理

微信小程序 localStorage 上限约 10MB。

| 缓存项 | 预估大小 | 保留策略 |
|--------|---------|---------|
| projects | ~2KB | 始终保留 |
| today_records | ~1KB/天 | 保留最近 7 天 |
| month_records | ~5KB/月 | 保留最近 3 个月 |
| user_profile | ~10KB（含 base64 头像） | 始终保留 |
| pending_ops | ~1KB/条 | 同步后清除 |
| **总计** | ~40KB | 远低于 10MB 上限 |

**自动清理**：

- 登出时清除该用户所有缓存
- 每次启动检查，删除 3 个月前的 month_records
- pending_ops 重试 3 次失败后丢弃并记录日志

## 冲突处理

本应用为个人单设备使用，冲突概率极低。

**策略**：Last-Write-Wins + 去重

- add 操作同步时，先查是否存在相同 `projectId + date` 的记录，有则跳过
- remove 操作同步时，目标不存在则静默成功
- 同步成功后全量拉取云端数据刷新本地缓存

## Store + 缓存结合设计

当前 Pinia Store（`stores/daka.js`）仅做内存缓存，页面刷新或 App 重启后数据丢失。将 Store 和 localStorage 缓存结合，形成**持久化 Store**，可以一举两得：

### 结合架构

```
┌─────────────────────────────────────────────────────┐
│                  Pinia Store（内存）                   │
│                                                      │
│  projects / todayRecords / pendingOps / syncStatus   │
│                                                      │
│  ┌──────────┐       ┌──────────┐                    │
│  │ 读/写     │       │ 状态管理  │                    │
│  │ 数据      │       │ dirty    │                    │
│  │          │       │ syncing  │                    │
│  └─────┬────┘       └──────────┘                    │
│        │ 自动持久化                                   │
│        ▼                                             │
│  ┌──────────────────────────────────┐               │
│  │        localStorage              │               │
│  │  (Store 变更时自动写入)            │               │
│  │  (App 启动时自动恢复)              │               │
│  └──────────────────────────────────┘               │
└─────────────────────────────────────────────────────┘
```

### 改造后的 Store

```js
// stores/daka.js

import { defineStore } from 'pinia'
import { ref, computed, watch } from 'vue'
import {
  getCachedProjects, setCachedProjects,
  getCachedTodayRecords, setCachedTodayRecords,
  getStoreKey, getLocal, setLocal,
} from '@/utils/local-store'
import { getAccountId } from '@/utils/auth'
import { formatDate } from '@/utils/date'

const CACHE_TTL = 5 * 60 * 1000

export const useDakaStore = defineStore('daka', () => {
  const projects = ref([])
  const todayRecords = ref([])
  const loading = ref(false)
  const pendingCount = ref(0)       // 待同步数量
  const isOffline = ref(false)      // 是否处于离线状态

  let _lastFetchTime = 0
  const _dirty = ref(false)

  // --- 从 localStorage 恢复 ---
  function restore() {
    const accountId = getAccountId()
    if (!accountId) return

    const cached = getCachedProjects(accountId)
    if (cached && cached.length > 0) {
      projects.value = cached
    }

    const today = formatDate(new Date())
    const cachedRecs = getCachedTodayRecords(accountId, today)
    if (cachedRecs) {
      todayRecords.value = cachedRecs
    }

    // 恢复 pending 计数
    const ops = getLocal(getStoreKey(accountId, 'pending', 'ops'))
    pendingCount.value = ops?.data?.length || 0
  }

  // --- 持久化到 localStorage ---
  function persist() {
    const accountId = getAccountId()
    if (!accountId) return

    setCachedProjects(accountId, projects.value)
    const today = formatDate(new Date())
    setCachedTodayRecords(accountId, today, todayRecords.value)
  }

  // --- 缓存管理 ---
  function isCacheValid() {
    return !_dirty.value && projects.value.length > 0 &&
           Date.now() - _lastFetchTime < CACHE_TTL
  }

  function markDirty() { _dirty.value = true }

  function markFresh() {
    _dirty.value = false
    _lastFetchTime = Date.now()
    persist()  // 云端数据拉取成功后持久化
  }

  // --- 数据操作（自动持久化） ---
  function setProjects(list) {
    projects.value = list
  }

  function setTodayRecords(list) {
    todayRecords.value = list
  }

  function addTodayRecord(record) {
    todayRecords.value.push(record)
    persist()  // 本地写操作后立即持久化
  }

  function removeTodayRecord(projectId) {
    todayRecords.value = todayRecords.value.filter(r => r.projectId !== projectId)
    persist()
  }

  function clear() {
    projects.value = []
    todayRecords.value = []
    _lastFetchTime = 0
    _dirty.value = true
    pendingCount.value = 0
  }

  return {
    projects, todayRecords, loading, pendingCount, isOffline,
    activeProjects: computed(() =>
      projects.value.filter(p => !p.archived).sort((a, b) => a.sortOrder - b.sortOrder)
    ),
    todayProgress: computed(() => {
      const total = projects.value.filter(p => !p.archived).length
      if (total === 0) return { done: 0, total: 0, percent: 0 }
      const done = todayRecords.value.length
      return { done, total, percent: Math.round((done / total) * 100) }
    }),
    restore, persist,
    isCacheValid, markDirty, markFresh,
    setProjects, setTodayRecords, addTodayRecord, removeTodayRecord, clear,
  }
})
```

### 关键变化

| 方面 | 现有 Store | 结合后的 Store |
|------|-----------|---------------|
| 数据来源 | 仅云端 | 启动时从 localStorage 恢复 |
| 数据持久化 | 无 | 写操作后自动持久化到 localStorage |
| 离线状态 | 无感知 | `isOffline` + `pendingCount` 驱动 UI |
| 页面刷新后 | 数据丢失，需重新请求 | 立即从本地恢复，无需等待云端 |
| pending 队列 | 不在 Store 中 | `pendingCount` 暴露给 UI |

### 使用方式

```js
// App.vue onLaunch
const dakaStore = useDakaStore()
dakaStore.restore()  // 启动时从 localStorage 恢复数据

// 首页 onShow
onShow(() => {
  // Store 已有数据（来自 localStorage），UI 立即渲染
  // 后台静默拉取云端最新数据
  if (!dakaStore.isCacheValid()) {
    loadData()  // 异步更新
  }
})
```

## 可行性评估

### 技术可行性：高

| 维度 | 评估 | 说明 |
|------|------|------|
| localStorage 容量 | 充足 | 预估 ~40KB，远低于微信 10MB 上限 |
| API 兼容 | 完全兼容 | `uni.setStorageSync` / `uni.getStorageSync` 全平台支持 |
| 数据一致性 | 可控 | 个人单设备，无多端冲突风险 |
| 改造范围 | 适中 | 主要改造 API 层和 Store，页面层改动小 |
| 性能影响 | 正向 | 本地读写 < 1ms，比云端请求快 100x |

### 风险与注意事项

#### 1. 数据丢失风险

| 风险场景 | 发生条件 | 影响 | 应对 |
|---------|---------|------|------|
| 用户清除小程序数据 | 主动操作 | 丢失未同步的离线打卡 | pending_ops 提示用户"有 N 条未同步" |
| 微信更新导致存储清空 | 极少发生 | 同上 | 同上 |
| localStorage 写入失败 | 存储已满（几乎不可能） | 降级为纯云端模式 | try/catch 保护 |

**结论**：风险极低。即使丢失，最多损失当天几条打卡记录。

#### 2. 数据一致性

| 场景 | 问题 | 处理 |
|------|------|------|
| 离线打卡后同步 | 可能与云端已有记录重复 | 同步前查重（`projectId + date`） |
| 离线取消后同步 | 云端可能已被其他操作影响 | remove 目标不存在则静默成功 |
| 跨天未同步 | 昨天的打卡今天才同步 | 以 record.date 为准，不受同步时间影响 |

**结论**：打卡记录以 `projectId + date` 为天然唯一键，重复和冲突都可安全处理。

#### 3. 批量同步的注意事项

| 注意点 | 说明 |
|--------|------|
| 顺序敏感 | 同一项目的 add 和 remove 必须按时间顺序执行 |
| 额度感知 | 同步过程中如果遇到额度限制，立即停止，不要浪费剩余额度 |
| 重试上限 | 单条操作最多重试 3 次，避免死循环 |
| 同步锁 | `_syncing` 标志防止并发同步 |
| 防抖 | 300ms 防抖，短时间内多次操作只触发一次同步 |

#### 4. 定时器管理

| 注意点 | 说明 |
|--------|------|
| App 进入后台 | 微信小程序后台运行有限制，定时器可能不触发 |
| 建议 | 不依赖定时器作为主要同步手段，改为 onShow 触发 |
| 内存泄漏 | App 退出/页面销毁时需清除定时器 |

#### 5. 边界场景

| 场景 | 预期行为 |
|------|---------|
| pending 队列很长（>100 条） | 分批同步，每批 10 条，避免单次请求太多 |
| 用户登出但有未同步数据 | 弹窗提醒"有 N 条记录未同步，是否放弃？" |
| 日期切换（0:00） | today_records 缓存 key 包含日期，自动分隔 |
| 首次安装无缓存 | 正常走云端请求，首次请求后建立缓存 |

#### 6. 版本号机制的注意事项

| 注意点 | 说明 |
|--------|------|
| 版本号递增时机 | 仅在写操作**云端同步成功**后递增，离线操作不递增 |
| 查询版本号的成本 | 每次 1 次 db 查询（查 user 文档），但远小于全量拉取的 2-3 次 |
| 版本号不一致时 | 必须全量拉取，不支持增量更新（增量方案复杂度过高） |
| 多个写操作并发 | 版本号可能跳跃（42→44），不影响逻辑（只判断是否一致） |
| user 文档不存在 | dataVersion 默认为 0，触发全量拉取 |
| 版本号 vs TTL | 版本号优先于 TTL。即使 TTL 未过期，版本号不一致也应刷新 |

| 维度 | 评估 | 说明 |
|------|------|------|
| localStorage 容量 | 充足 | 预估 ~40KB，远低于微信 10MB 上限 |
| API 兼容 | 完全兼容 | `uni.setStorageSync` / `uni.getStorageSync` 全平台支持 |
