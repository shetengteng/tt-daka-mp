# 离线缓存设计方案（Local-First）

## 背景

EMAS 免费版每日有请求额度限制（`PrePayResourceExhausted`），超限后所有数据库操作失败。当前系统在云服务不可用时，用户无法打卡、无法查看数据。

**核心思路**：所有操作先写本地 localStorage，UI 立即响应；云端同步作为后台异步任务，可批量合并、延迟执行，大幅减少 API 调用次数。

## 缓存触发时机

### 写缓存时机

| 时机 | 写入什么 | 触发方 |
|------|---------|--------|
| 云端查询成功 | API 结果 → Store → localStorage | Store.setList() / setTodayRecords() |
| 本地写操作 | 增量更新 Store → localStorage | Store.addTodayRecord() / removeTodayRecord() |
| 同步成功 | 更新 meta.lastSync + dataVersion | SyncManager |

### 读缓存时机

| 时机 | 读什么 | 条件 |
|------|--------|------|
| App 启动 | Store.restore() 从 localStorage 恢复 | 始终执行 |
| 页面 onShow | Store 中已有数据，直接渲染 | isCacheValid() 判断 |
| 日历切换月份 | month_{YYYY-MM} | 本地有则直接用 |
| 云端请求失败 | Store 已有缓存可用 | API catch 中降级 |

### 同步触发时机

| 时机 | 行为 | 说明 |
|------|------|------|
| App onLaunch/onShow | 检查队列，有则同步 | 启动或回到前台 |
| 下拉刷新 | 强制同步 + 全量拉取 | 用户主动刷新 |
| 写操作后（延迟） | 300ms 后尝试同步 | 防抖，合并短时间多次操作 |
| 定时器 | 30s 轮询检查 | 静默后台同步 |

## Local-First 写入策略（节省 API 次数）

### 核心思路

传统模式：每次打卡 → 立即调云端 → 1 次 API  
Local-First：N 次打卡 → 本地即时响应 → 批量同步 → 可能只需 1-2 次 API

### 批量合并规则

```
场景1：对同一项目 先打卡后取消
  add(proj_A, 04-05) + remove(proj_A, 04-05) → 抵消，无需同步

场景2：同一时间段多个项目打卡
  add(proj_A) + add(proj_B) + add(proj_C) → 保持 3 条，无法合并（不同文档）

场景3：补打卡多天
  add(proj_A, 04-01) + add(proj_A, 04-02) + add(proj_A, 04-03) → 保持 3 条
```

**合并算法**：

```js
function mergeOps(ops) {
  const cancelMap = new Map()

  for (let i = ops.length - 1; i >= 0; i--) {
    const op = ops[i]
    // remove 操作的 projectId/date 在 where 或 data 中
    const projectId = op.data?.projectId || op.where?.projectId
    const date = op.data?.date || op.where?.date || ''
    const key = `${op.collection}_${projectId}_${date}`

    if (op.op === 'remove') {
      cancelMap.set(key, i)
    } else if (op.op === 'add' && cancelMap.has(key)) {
      ops[i].merged = true
      ops[cancelMap.get(key)].merged = true
      cancelMap.delete(key)
    }
  }

  return ops.filter(op => !op.merged)
}
```

### 批量插入（insertMany）

EMAS 支持 `insertMany` 批量插入，可以将多条 add 操作合并为**一次 API 调用**。

**封装接口**：

```js
// cloud-emas/database/database.js 新增方法

async addBatch(dataList) {
  try {
    const collection = await this._getCollection()
    const res = await collection.insertMany(dataList)
    return {
      count: res.result ? Object.keys(res.result).length : 0,
      ids: res.result || {},
      errMsg: 'ok',
    }
  } catch (e) {
    handleEmasError(e, `批量新增 ${this._collectionName}`)
  }
}
```

**同步时的批量合并**：

```js
// SyncManager 中按 collection + op 分组
function groupOps(ops) {
  const addGroups = {}   // { collection: [data1, data2, ...] }
  const others = []      // remove / update 无法批量

  for (const op of ops) {
    if (op.op === 'add') {
      if (!addGroups[op.collection]) addGroups[op.collection] = []
      addGroups[op.collection].push({ opId: op.id, data: op.data })
    } else {
      others.push(op)
    }
  }

  return { addGroups, others }
}

// 批量同步
async function syncBatch(accountId) {
  const ops = mergeOps(getPendingOps(accountId))
  const { addGroups, others } = groupOps(ops)

  // 1. 批量插入（每组 1 次 API）
  for (const [collection, items] of Object.entries(addGroups)) {
    try {
      await db.collection(collection).addBatch(items.map(i => i.data))
      items.forEach(i => removePendingOp(accountId, i.opId))
    } catch (e) {
      if (isQuotaError(e)) return
    }
  }

  // 2. 执行 remove / update（按条执行，EMAS 不支持批量删除/更新不同条件）
  for (const op of others) {
    // ...同 SyncManager 中的 otherOps 逻辑
  }
}
```

### 效果对比

| 场景 | 传统模式 | Local-First（批量同步） | 节省 |
|------|---------|----------------------|------|
| 打卡 3 个项目 | 3 次 API | **1 次**（insertMany） | 66% |
| 打卡后反悔取消 | 2 次 API | 0 次（抵消） | 100% |
| 日内反复打卡取消 5 次 | 5 次 API | 0-1 次 | 80-100% |
| 补打卡一周 7 天 | 7 次 API | **1 次**（insertMany） | 86% |
| 额度用完后打卡 10 次 | 失败 | 次日 **1 次** 批量同步 | 90% |

## 版本号机制（减少无效拉取）

### 问题

后台静默拉取云端数据时，即使数据没有变化也会消耗 API 调用次数。需要一种机制判断"云端数据是否有更新"，没有更新则跳过拉取。

### 方案

在 `dk-users` 表中增加 `dataVersion` 字段，使用**时间戳**作为乐观锁。每次写操作同步到云端时，将 `dataVersion` 设为 `Date.now()`。本地缓存记录上次同步的时间戳，对比时只需判断云端版本是否比本地**更大**，更大则需要刷新。

**使用时间戳而非递增整数的优势**：
- 无需先读后写（不用先查当前值再 +1），直接 `update({ dataVersion: Date.now() })`，省 1 次读操作
- 天然有序，支持"比大小"的乐观锁语义
- 多次写操作产生的版本号不会冲突

### 数据结构变更

```js
// dk-users 表增加字段
{
  // ...原有字段
  dataVersion: 1712300000000,    // 时间戳，每次写操作后更新为 Date.now()
}

// localStorage 缓存增加版本号
{
  data: [...],
  timestamp: 1712300000000,
  source: 'cloud',
  dataVersion: 1712300000000,   // 记录缓存对应的云端版本时间戳
}
```

### 版本号检查接口

```js
// utils/version-check.js

import { db, COLLECTIONS } from '@/cloud-emas/database/database'
import { getLocal, setLocal, getStoreKey } from './local-store'

export async function checkDataVersion(accountId) {
  const versionKey = getStoreKey(accountId, 'meta', 'dataVersion')
  const localVersion = getLocal(versionKey)?.data || 0

  try {
    // 使用 field() 只拉取 dataVersion 字段，避免传输 base64 头像等大数据
    const res = await db.collection(COLLECTIONS.USERS)
      .where({ accountId })
      .field({ dataVersion: 1 })
      .get()
    const user = (res.data || [])[0]
    const cloudVersion = user?.dataVersion || 0

    if (cloudVersion <= localVersion && localVersion > 0) {
      return { needRefresh: false, version: localVersion }
    }

    return { needRefresh: true, version: cloudVersion }
  } catch (e) {
    return { needRefresh: false, version: localVersion }
  }
}

export function updateLocalVersion(accountId, version) {
  const versionKey = getStoreKey(accountId, 'meta', 'dataVersion')
  setLocal(versionKey, version)
}

export async function touchCloudVersion(accountId) {
  try {
    const now = Date.now()
    await db.collection(COLLECTIONS.USERS)
      .where({ accountId })
      .update({ dataVersion: now })
    return now
  } catch (e) {
    // 版本号更新失败不影响主流程
  }
}
```

**对比旧方案（递增整数）**：

| 维度 | 递增整数 | 时间戳乐观锁 |
|------|---------|-------------|
| 写入 | 先读当前值 → +1 → 写入（2 次 API） | 直接写 Date.now()（1 次 API） |
| 对比逻辑 | `cloudVersion === localVersion` | `cloudVersion <= localVersion` |
| 并发安全 | 可能冲突（两次 +1 变成相同值） | 天然安全（时间戳几乎不会相同） |
| 可读性 | 42, 43, 44... | 1712300000000（可转日期） |

### 版本号使用场景

| 场景 | 行为 | API 调用 |
|------|------|---------|
| 页面 onShow（有缓存） | 查 user.dataVersion → 对比 → 跳过/拉取 | 1 次（仅查 version） |
| 页面 onShow（无缓存） | 直接全量拉取 | 2-3 次 |
| 打卡同步成功 | `dataVersion = Date.now()` | 1 次 update（无需先 read） |
| 下拉刷新 | 强制拉取，不检查版本 | 正常次数 |

### 效果

| 场景 | 无版本号 | 有版本号 | 节省 |
|------|---------|---------|------|
| 切换 Tab 4 次（数据无变化） | 8 次 API | 4 次（每次 1 查 version） | 50% |
| App 启动恢复（数据无变化） | 2-3 次 | 1 次 | 50-66% |
| 一天内切 Tab 20 次 | 40 次 | 20 次 | 50% |
| 写操作更新版本号 | 2 次（read + write） | 1 次（直接 write） | 50% |

### 注意事项

| 注意点 | 说明 |
|--------|------|
| 时间戳精度 | `Date.now()` 毫秒级，同一毫秒内多次写入值相同，但概率极低 |
| 首次使用 | `user.dataVersion` 不存在时视为 0，触发全量拉取 |
| 离线操作 | 本地写操作不更新 cloudVersion，同步成功后才更新 |
| 时钟偏移 | 个人单设备，不存在多设备时钟不一致问题 |

```js
// utils/local-store.js
// 不含业务逻辑，只提供原子读写操作

const PREFIX = 'dk_'

export function getStoreKey(accountId, ...parts) {
  return `${PREFIX}${accountId}_${parts.join('_')}`
}

export function setLocal(key, data) {
  try {
    uni.setStorageSync(key, JSON.stringify({
      data,
      timestamp: Date.now(),
      source: 'local',
    }))
  } catch (e) {
    console.warn('[LocalStore] 写入失败:', key, e)
  }
}

export function getLocal(key) {
  try {
    const raw = uni.getStorageSync(key)
    if (!raw) return null
    return JSON.parse(raw)
  } catch (e) {
    return null
  }
}

export function removeLocal(key) {
  try {
    uni.removeStorageSync(key)
  } catch (e) {}
}

export function clearUserCache(accountId) {
  try {
    const info = uni.getStorageInfoSync()
    const prefix = `${PREFIX}${accountId}_`
    ;(info.keys || []).forEach(key => {
      if (key.startsWith(prefix)) uni.removeStorageSync(key)
    })
  } catch (e) {}
}
```

### 模块二：PendingOps（待同步队列）

```js
// utils/pending-ops.js

import { getLocal, setLocal, getStoreKey } from './local-store'

function _getKey(accountId) {
  return getStoreKey(accountId, 'pending', 'ops')
}

export function getPendingOps(accountId) {
  const entry = getLocal(_getKey(accountId))
  return entry?.data || []
}

export function addPendingOp(accountId, op) {
  const ops = getPendingOps(accountId)
  ops.push({
    ...op,
    id: `op_${Date.now()}_${Math.random().toString(36).slice(2, 6)}`,
    timestamp: Date.now(),
    retryCount: 0,
    merged: false,
  })
  setLocal(_getKey(accountId), ops)
}

export function removePendingOp(accountId, opId) {
  const ops = getPendingOps(accountId).filter(o => o.id !== opId)
  setLocal(_getKey(accountId), ops)
}

export function savePendingOps(accountId, ops) {
  setLocal(_getKey(accountId), ops)
}

export function clearPendingOps(accountId) {
  setLocal(_getKey(accountId), [])
}

export function getPendingCount(accountId) {
  return getPendingOps(accountId).length
}
```

### 模块三：SyncManager（同步管理器）

```js
// utils/sync-manager.js

import { getPendingOps, removePendingOp } from './pending-ops'
import { db, COLLECTIONS } from '@/cloud-emas/database/database'
import { isQuotaError } from '@/cloud-emas/database/error'

let _syncing = false
let _syncTimer = null

export function startSyncLoop() {
  if (_syncTimer) return
  _syncTimer = setInterval(() => {
    syncPendingOps()
  }, 30 * 1000)
}

export function stopSyncLoop() {
  if (_syncTimer) {
    clearInterval(_syncTimer)
    _syncTimer = null
  }
}

export async function syncPendingOps(accountId) {
  if (_syncing || !accountId) return { synced: 0, remaining: 0 }

  let ops = getPendingOps(accountId)
  if (ops.length === 0) return { synced: 0, remaining: 0 }

  // 1. 合并抵消操作（add + remove 同一记录 → 抵消）
  ops = mergeOps(ops)

  _syncing = true
  let syncedCount = 0

  // 2. 按 collection + op 分组，add 操作走 insertMany 批量插入
  const addGroups = {}
  const otherOps = []

  for (const op of ops) {
    if (op.op === 'add') {
      const key = op.collection
      if (!addGroups[key]) addGroups[key] = []
      addGroups[key].push(op)
    } else {
      otherOps.push(op)
    }
  }

  // 3. 批量插入（每个 collection 1 次 API）
  for (const [collection, groupedOps] of Object.entries(addGroups)) {
    try {
      await db.collection(collection).addBatch(groupedOps.map(op => op.data))
      groupedOps.forEach(op => removePendingOp(accountId, op.id))
      syncedCount += groupedOps.length
    } catch (e) {
      if (isQuotaError(e)) {
        _syncing = false
        return { synced: syncedCount, remaining: getPendingOps(accountId).length }
      }
      // 批量失败 → 超过重试次数的逐个移除，其余保留待下次重试
      // 注：retryCount 需通过 updatePendingOp 持久化回 localStorage
      groupedOps.forEach(op => {
        op.retryCount++
        if (op.retryCount >= 3) removePendingOp(accountId, op.id)
      })
      savePendingOps(accountId, getPendingOps(accountId))
    }
  }

  // 4. 执行 remove / update（不同条件，无法合并为一次调用）
  for (const op of otherOps) {
    try {
      if (op.op === 'remove') {
        await db.collection(op.collection).where(op.where).remove()
      } else if (op.op === 'update') {
        await db.collection(op.collection).doc(op.docId).update(op.data)
      }
      removePendingOp(accountId, op.id)
      syncedCount++
    } catch (e) {
      if (isQuotaError(e)) break
      op.retryCount++
      if (op.retryCount >= 3) removePendingOp(accountId, op.id)
      break
    }
  }

  // 5. 同步成功后更新云端版本号
  if (syncedCount > 0) {
    await touchCloudVersion(accountId)
    uni.showToast({ title: `已同步 ${syncedCount} 条离线记录`, icon: 'none' })
  }

  _syncing = false
  return { synced: syncedCount, remaining: getPendingOps(accountId).length }
}

function mergeOps(ops) {
  const cancelMap = new Map()

  for (let i = ops.length - 1; i >= 0; i--) {
    const op = ops[i]
    if (op.collection !== COLLECTIONS.RECORDS) continue

    const key = `${op.data?.projectId}_${op.data?.date || op.where?.date}`

    if (op.op === 'remove') {
      cancelMap.set(key, i)
    } else if (op.op === 'add' && cancelMap.has(key)) {
      ops[i].merged = true
      ops[cancelMap.get(key)].merged = true
      cancelMap.delete(key)
    }
  }

  return ops.filter(op => !op.merged)
}
```

## localStorage 容量管理

微信小程序 localStorage 上限约 10MB。

| 缓存项 | 预估大小 | 保留策略 |
|--------|---------|---------|
| projects | ~2KB | 始终保留 |
| today_records | ~1KB/天 | 保留最近 7 天 |
| month_records | ~5KB/月 | 保留最近 3 个月 |
| user_profile | ~10KB（含 base64 头像） | 始终保留 |
| pending_ops | ~1KB/条 | 同步后清除 |
| **总计** | ~40KB | 远低于 10MB 上限 |

**自动清理**：

- 登出时清除该用户所有缓存
- 每次启动检查，删除 3 个月前的 month_records
- pending_ops 重试 3 次失败后丢弃并记录日志

## 冲突处理

本应用为个人单设备使用，冲突概率极低。

**策略**：Last-Write-Wins + 去重

- add 操作同步时，先查是否存在相同 `projectId + date` 的记录，有则跳过
- remove 操作同步时，目标不存在则静默成功
- 同步成功后全量拉取云端数据刷新本地缓存

## Store 架构设计（Store 统管缓存）

### 设计原则

- **Store 是唯一的数据入口**：API 层查询成功后交给 Store，Store 内部完成内存 + localStorage 双写
- **API 层不直接调 localStorage**：API 层只与 Store 交互（set/get），由 Store 决定何时持久化
- **`local-store.js` 是纯存储工具**：只提供 `setLocal` / `getLocal` / `removeLocal` 等原子操作，不含业务快捷方法
- **拆分 Store**：按数据领域拆分为 `projectStore` 和 `recordStore`，各自管理各自的缓存

### Store 拆分方案

#### stores/project.js（项目 Store）

```js
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { getAccountId } from '@/utils/auth'
import { setLocal, getLocal, getStoreKey } from '@/utils/local-store'

const CACHE_TTL = 5 * 60 * 1000
const CACHE_KEY = 'cache_projects'

export const useProjectStore = defineStore('project', () => {
  const list = ref([])
  const loading = ref(false)

  let _lastFetchTime = 0
  const _dirty = ref(false)

  const activeList = computed(() =>
    list.value.filter(p => !p.archived).sort((a, b) => a.sortOrder - b.sortOrder)
  )

  function restore() {
    const accountId = getAccountId()
    if (!accountId) return
    const entry = getLocal(getStoreKey(accountId, CACHE_KEY))
    if (entry?.data?.length > 0) {
      list.value = entry.data
    }
  }

  function persist() {
    const accountId = getAccountId()
    if (!accountId) return
    setLocal(getStoreKey(accountId, CACHE_KEY), list.value)
  }

  function isCacheValid() {
    return !_dirty.value && list.value.length > 0 &&
           Date.now() - _lastFetchTime < CACHE_TTL
  }

  function markDirty() { _dirty.value = true }

  function markFresh() {
    _dirty.value = false
    _lastFetchTime = Date.now()
    persist()
  }

  function setList(data) {
    list.value = data
    persist()
  }

  function clear() {
    list.value = []
    _lastFetchTime = 0
    _dirty.value = true
  }

  return {
    list, loading, activeList,
    restore, persist, isCacheValid, markDirty, markFresh,
    setList, clear,
  }
})
```

#### stores/record.js（记录 Store）

```js
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { getAccountId } from '@/utils/auth'
import { formatDate } from '@/utils/date'
import { setLocal, getLocal, getStoreKey } from '@/utils/local-store'
import { useProjectStore } from './project'

const CACHE_TTL = 5 * 60 * 1000

export const useRecordStore = defineStore('record', () => {
  const todayRecords = ref([])
  const pendingCount = ref(0)
  const isOffline = ref(false)

  let _lastFetchTime = 0
  const _dirty = ref(false)

  const todayProgress = computed(() => {
    const projectStore = useProjectStore()
    const total = projectStore.activeList.length
    if (total === 0) return { done: 0, total: 0, percent: 0 }
    const done = todayRecords.value.length
    return { done, total, percent: Math.round((done / total) * 100) }
  })

  function _todayCacheKey() {
    const accountId = getAccountId()
    const today = formatDate(new Date())
    return accountId ? getStoreKey(accountId, 'cache', `today_${today}`) : null
  }

  function restore() {
    const key = _todayCacheKey()
    if (!key) return
    const entry = getLocal(key)
    if (entry?.data) {
      todayRecords.value = entry.data
    }

    const accountId = getAccountId()
    const ops = getLocal(getStoreKey(accountId, 'pending', 'ops'))
    pendingCount.value = ops?.data?.length || 0
  }

  function persist() {
    const key = _todayCacheKey()
    if (!key) return
    setLocal(key, todayRecords.value)
  }

  function isCacheValid() {
    return !_dirty.value && Date.now() - _lastFetchTime < CACHE_TTL
  }

  function markDirty() { _dirty.value = true }

  function markFresh() {
    _dirty.value = false
    _lastFetchTime = Date.now()
    persist()
  }

  function setTodayRecords(list) {
    todayRecords.value = list
    persist()
  }

  function addTodayRecord(record) {
    todayRecords.value.push(record)
    persist()
  }

  function removeTodayRecord(projectId) {
    todayRecords.value = todayRecords.value.filter(r => r.projectId !== projectId)
    persist()
  }

  function clear() {
    todayRecords.value = []
    _lastFetchTime = 0
    _dirty.value = true
    pendingCount.value = 0
  }

  return {
    todayRecords, pendingCount, isOffline, todayProgress,
    restore, persist, isCacheValid, markDirty, markFresh,
    setTodayRecords, addTodayRecord, removeTodayRecord, clear,
  }
})
```

## 可行性评估

### 技术可行性：高

| 维度 | 评估 | 说明 |
|------|------|------|
| localStorage 容量 | 充足 | 预估 ~40KB，远低于微信 10MB 上限 |
| API 兼容 | 完全兼容 | `uni.setStorageSync` / `uni.getStorageSync` 全平台支持 |
| 数据一致性 | 可控 | 个人单设备，无多端冲突风险 |
| 改造范围 | 适中 | 拆分 Store + 改造 API 层，页面层引用替换 |
| 性能影响 | 正向 | 本地读写 < 1ms，比云端请求快 100x |

### 风险与注意事项

#### 1. 数据丢失风险

| 风险场景 | 发生条件 | 影响 | 应对 |
|---------|---------|------|------|
| 用户清除小程序数据 | 主动操作 | 丢失未同步的离线打卡 | pending_ops 提示用户"有 N 条未同步" |
| 微信更新导致存储清空 | 极少发生 | 同上 | 同上 |
| localStorage 写入失败 | 存储已满（几乎不可能） | 降级为纯云端模式 | try/catch 保护 |

**结论**：风险极低。即使丢失，最多损失当天几条打卡记录。

#### 2. 数据一致性

| 场景 | 问题 | 处理 |
|------|------|------|
| 离线打卡后同步 | 可能与云端已有记录重复 | 同步前查重（`projectId + date`） |
| 离线取消后同步 | 云端可能已被其他操作影响 | remove 目标不存在则静默成功 |
| 跨天未同步 | 昨天的打卡今天才同步 | 以 record.date 为准，不受同步时间影响 |

**结论**：打卡记录以 `projectId + date` 为天然唯一键，重复和冲突都可安全处理。

#### 3. 批量同步的注意事项

| 注意点 | 说明 |
|--------|------|
| 顺序敏感 | 同一项目的 add 和 remove 必须按时间顺序执行 |
| 额度感知 | 同步过程中如果遇到额度限制，立即停止，不要浪费剩余额度 |
| 重试上限 | 单条操作最多重试 3 次，避免死循环 |
| 同步锁 | `_syncing` 标志防止并发同步 |
| 防抖 | 300ms 防抖，短时间内多次操作只触发一次同步 |

#### 4. 定时器管理

| 注意点 | 说明 |
|--------|------|
| App 进入后台 | 微信小程序后台运行有限制，定时器可能不触发 |
| 建议 | 不依赖定时器作为主要同步手段，改为 onShow 触发 |
| 内存泄漏 | App 退出/页面销毁时需清除定时器 |

#### 5. 边界场景

| 场景 | 预期行为 |
|------|---------|
| pending 队列很长（>100 条） | 分批同步，每批 10 条，避免单次请求太多 |
| 用户登出但有未同步数据 | 弹窗提醒"有 N 条记录未同步，是否放弃？" |
| 日期切换（0:00） | today_records 缓存 key 包含日期，自动分隔 |
| 首次安装无缓存 | 正常走云端请求，首次请求后建立缓存 |

#### 6. 版本号机制的注意事项

| 注意点 | 说明 |
|--------|------|
| 版本号更新时机 | 仅在写操作**云端同步成功**后 `touchCloudVersion()`，离线操作不更新 |
| 查询版本号的成本 | 每次 1 次 db 查询，使用 `.field({ dataVersion: 1 })` 避免拉取大文档 |
| 版本号更大时 | 必须全量拉取，不支持增量更新（增量方案复杂度过高） |
| 多次同步写入 | 每次 `Date.now()` 值不同，天然有序，不会冲突 |
| user 文档不存在 | dataVersion 默认为 0，触发全量拉取 |
| 版本号 vs TTL | 版本号优先于 TTL。即使 TTL 未过期，云端版本更大也应刷新 |

### 完整可行性总结

| 维度 | 评估 | 说明 |
|------|------|------|
| localStorage 容量 | 充足 | 预估 ~40KB，远低于微信 10MB 上限 |
| API 兼容 | 完全兼容 | `uni.setStorageSync` / `uni.getStorageSync` 全平台支持 |
| 数据一致性 | 可控 | 个人单设备 + 版本号机制，可靠性高 |
| 改造范围 | 适中 | 拆分 Store + 改造 API 层，页面层引用替换 |
| 性能影响 | 正向 | 本地读写 < 1ms，比云端请求快 100x |
| 版本号检查 | 低成本 | `.field({ dataVersion: 1 })` 仅拉版本号，替代全量查询 |

### 收益总结

| 指标 | 当前 | Local-First + 版本号 | + 批量操作 | 最终改善 |
|------|------|---------------------|-----------|---------|
| 云服务不可用时体验 | 完全不可用 | 正常打卡 | 正常打卡 | 根本性改善 |
| 每日 API 调用次数 | 170+ | 15-30 | **10-20** | 减少 85-95% |
| 打卡 N 个项目 | N 次 API | N 次 | **1 次**（insertMany） | 减少 60-90% |
| Tab 切换（数据无变化） | 2-3 次/次 | 1 次（查版本号） | 1 次 | 减少 50-66% |
| 页面加载速度 | 等待云端 | 本地即时 | 本地即时 | 接近 0 延迟 |
| 数据安全性 | 仅云端 | 云端+本地 | 云端+本地 | 提升 |

### 不建议做的

| 事项 | 原因 |
|------|------|
| 离线支持项目管理（增删改） | 复杂度高，收益低（低频操作） |
| 离线支持用户信息修改 | 头像 base64 数据大，占用 localStorage |
| 多设备同步 | 当前为个人单设备，无需考虑 |
| 全量数据本地化 | 历史数据量增长后 localStorage 不够用 |
| 增量数据同步 | 复杂度远高于版本号 + 全量拉取方案 |
